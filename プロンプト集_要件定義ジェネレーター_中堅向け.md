# プロンプト集【中堅向け】｜要件定義ジェネレーターで“レビューに耐える”設計書を作る

> 🐰🦺 面白きこともなき世を面白く（高杉晋作）
> 「うさうさ開発フロア PRO（要件定義＆詳細設計ジェネレーター）」に貼り戻して使う、**中堅エンジニア向け**のAIプロンプト集です。前提・トレードオフ・定量化・監査・後輩指導まで踏み込みます。

## この集の前提

基礎（FR/NFR/GWT/ADR/トレーサビリティの意味）は既知とします。狙いは **「速く書く」ではなく「レビューと運用に耐える密度で書く」** こと。AIは下書き役、最終判断は人、という運用を崩しません。

> 💡 中堅の腕の見せ所は、**“決めていないこと”を可視化して潰す**ことと、**トレードオフを言語化して残す**ことです。

## フェーズ1：要件の質を上げる

### 1-1. 背景の“暗黙の前提”を炙り出す

> **ねらい**：背景に潜む未検証の前提・思い込みを表に出します。

```text
あなたは要件レビューを担うシニアエンジニアです。
次の背景・目的に潜む「暗黙の前提」と「検証すべき仮説」を洗い出してください。

# 背景・目的
{背景・目的を貼り付け}

# 出力
- 「明文化されていない前提」を列挙し、それぞれ崩れたときの影響を1行で。
- 「事実」と「仮説（未検証）」を分け、仮説には検証方法と確認先（役職名）を添える。
- 数値・前提に根拠がない箇所を指摘（ウソなく。憶測には「※推測」）。
```

### 1-2. スコープの“境界線”を引き直す

```text
次のスコープ案を、境界が曖昧な箇所を中心にレビューしてください。

# スコープ案（やる/やらない）
{スコープを貼り付け}

# 出力
- 「やる/やらない」のどちらとも取れるグレーゾーンを抽出し、決め方を提案。
- スコープ外なのに依存している外部要因（前提リスク）を指摘。
- フェーズ分割の提案（MVP / 次フェーズ）と、その分割の判断軸。
```

### 1-3. FRの粒度・優先度を是正する

```text
次のFR一覧を、粒度と優先度の観点でレビュー・是正してください。

# FR一覧
{FRを貼り付け}

# 出力
- 粒度が粗い（複合的な）FRを分割案として提示。
- 優先度（MoSCoW）の付け方に異論があれば、根拠つきで再提案。
- 受入基準(GWT)が検証不能・主観的なものを指摘し、計測可能な形に書き直す。
- 相互に矛盾・重複するFRがあれば指摘。
```

### 1-4. NFRを“測れる目標値”まで落とす

> **ねらい**：「速い」「安全」を、合否判定できる数値・条件にします。

```text
次のシステムのNFRを、合否判定できる目標値まで具体化してください。

# システム概要 / 想定負荷
{概要・利用規模・ピーク等}

# 出力
- カテゴリ別（性能/可用性/セキュリティ/保守性/運用/互換性）。
- 各項目を「指標 / 目標値 / 測定方法」で定義（例：p95レイテンシ / ○ms以下 / 負荷試験で測定）。
- 目標値の根拠（要件・SLA・類似実績）を事実として明示。根拠が無いものは「要合意（誰と）」。
- トレードオフ（例：強整合性と可用性）が生じる項目に注意書き。
```

## フェーズ2：設計判断とアーキテクチャ

### 2-1. ADRをトレードオフ込みで書く

```text
次の設計判断について、トレードオフを明示したADRを書いてください。

# 判断対象 / 制約
{判断対象と制約}

# 出力
- 「論点 / 決定 / 理由 / 検討した代替案 / 却下理由」。
- 評価軸（性能・コスト・運用負荷・学習コスト・可逆性 等）で各案を比較した表。
- 「この決定を将来見直す条件（トリガー）」を1つ明記。
- 理由は事実（測定・要件）と所感（チーム事情）を分離。
```

### 2-2. アーキテクチャ選定を比較評価する

```text
次の要件に対し、現実的なアーキテクチャ案を2〜3つ挙げ、比較評価してください。

# 主要FR / 主要NFR
{FR・NFRの要点を貼り付け}

# 出力
- 各案の構成図（テキストツリー）と、満たすNFR・割り切るNFR。
- 比較表（評価軸：性能 / スケール / 複雑度 / 運用 / コスト / 可逆性）。
- 推奨案と、その推奨が崩れる条件（前提が変わったら別案、という分岐）。
```

### 2-3. 変更影響範囲（インパクト）を分析する

```text
次の変更について、影響範囲を分析してください。

# 変更内容
{変更したいこと}

# 既存の設計書
{関連部分のFR/モジュール/データ/APIを貼り付け}

# 出力
- 影響を受けるFR・モジュール・データ・APIを列挙（直接/間接を区別）。
- 後方互換性に関わる箇所と、移行手順の論点。
- 必要な追加テスト観点。
- 見落としやすい横断的影響（ログ・権限・性能）への注意。
```

## フェーズ3：詳細設計の堅牢性

### 3-1. データ整合・同時実行の落とし穴を洗う

```text
次のデータ設計について、整合性と同時実行の観点でレビューしてください。

# データ設計
{識別子/表/型/制約 を貼り付け}

# 出力
- 競合・二重更新・部分失敗が起きうる箇所と対策（楽観/悲観ロック、トランザクション境界）。
- 一意制約・外部キー・NULL可否の妥当性。
- 整合性レベル（強/結果整合）の選択とその理由。
- マイグレーション時の注意（既存データ・ダウンタイム）。
```

### 3-2. API契約を厳密化する

```text
次のAPIを、契約（コントラクト）として厳密化してください。

# API案
{メソッド/パス/req/res を貼り付け}

# 出力
- 各エンドポイントの正常系＋主要エラー（4xx/5xx）のステータスとボディ。
- 冪等性・ページング・バージョニング・レート制限の方針。
- 認証/認可の前提と、権限不足時の挙動。
- 後方互換を壊す変更の定義（破壊的変更チェック）。
```

### 3-3. 例外・ログ・可観測性を設計する

```text
次の処理群について、例外・ログ・可観測性を設計してください。

# 対象処理
{処理の概要}

# 出力
- 「事象 / 捕捉位置 / 利用者への扱い / ログ出力（レベル・含める情報・含めない情報）」の表。
- 握りつぶし・例外の握り直しになっていないかの指摘。
- 障害解析に必要な相関ID・メトリクス・アラート条件。
- 個人情報・秘密情報をログに出さないためのルール。
```

## フェーズ4：トレーサビリティ監査とレビュー

### 4-1. トレーサビリティを監査する

```text
次の「FR」と「テスト観点」を突き合わせ、トレーサビリティを監査してください。

# FR一覧
{FRを貼り付け}

# テスト観点（対応FR付き）
{テスト観点を貼り付け}

# 出力
- 未カバーFR（特にMust）を最優先で列挙。
- 存在しないFRを参照しているテストを列挙。
- 1テストに紐づきすぎ/曖昧な対応の指摘。
- 受入基準(GWT)に対し、正常・異常・境界のテストが揃っているかの充足度評価。
- 事実ベースで指摘し、推測には「※推測」を付与。
```

### 4-2. 辛口セルフレビュー（リリース判定）

```text
あなたは厳しめのアーキテクトです。次の設計書を、リリース可否の観点でレビューしてください。

# 設計書
{ツールの .md または JSON 出力を貼り付け}

# 出力
- 「曖昧 / 抜け / 検証不能 / 矛盾」を重要度順に。各指摘に改善案を1つ。
- リリースブロッカー（直さないと出せない）と、改善余地（出せるが要対応）を分ける。
- NFRが定量化されているか、トレーサビリティが閉じているかを判定。
- 過剰設計（YAGNI違反）の指摘も含める。良い点も3つは挙げる。
```

### 4-3. AI出力を検証・反証する

> **ねらい**：AIが自信満々に出した設計を、鵜呑みにせず叩きます。

```text
次は、AIが生成した設計案です。これを批判的に検証してください。

# AI生成の設計案
{貼り付け}

# 出力
- 事実誤認・古い前提・根拠のない一般化を指摘。
- 「もっともらしいが検証されていない主張」を抽出。
- 反証となりうるケース（この設計が破綻する条件）を提示。
- 採用してよい部分・要修正の部分・捨てるべき部分に仕分け。
```

## フェーズ5：後輩の設計書を指導する

### 5-1. 新人の設計書をメンタリング視点でレビューする

```text
あなたは新人を育てる中堅エンジニアです。
新人が書いた設計書を、指導の観点でレビューしてください。

# 新人の設計書
{貼り付け}

# 出力
- 「直すべき点」と「なぜそう書くと良いか」をセットで（手本を示す）。
- つぶしすぎない。良い着眼点は具体的に褒める。
- 本人が次に意識すべき“1つの観点”に絞って助言。
- 指摘は事実ベース。好みの問題は「※好みの範囲」と明記。
```

### 5-2. レビュー指摘を“伝わる言葉”に整える

```text
次のレビュー指摘を、相手が前向きに受け取れる言い回しに整えてください。

# 元の指摘（箇条書きでOK）
{指摘内容}

# 出力
- 事実→影響→提案、の順で1件ずつ。
- 断定しすぎず、確認を促す形（「〜という理解で合っていますか？」）。
- 役職名で記述、固有名は出さない。ですます調。
```

## 効かせ方：品質ゲートと効率化

- **役割と評価軸を先に固定**：「あなたは〜。評価軸は性能/コスト/運用/可逆性」と置くと、出力がブレません。
- **“決めていないこと”を必ず吐かせる**：プロンプト末尾に「未決事項と確認先（役職名）を最後に列挙」を足す。
- **トレードオフを言語化させる**：「割り切ったNFL/NFRと、その理由」を必ず書かせる。
- **二段構え**：生成（フェーズ1〜3）→ 監査・反証（4-1/4-3）を別プロンプトで回すと、品質が安定します。
- **出力形式を固定**：表・項目名を指定し、ツールのフォームへ機械的に貼り戻せる粒度に。

## 品質チェックリスト（厳しめ）

```text
レビュー前に、次がすべて満たせているか確認してください。

□ すべてのMust FRに、検証可能なGWT受入基準があるか？
□ すべてのFRが、最低1つのテスト観点でカバーされているか（帯が緑）？
□ 存在しないFRを参照するテストが無いか？
□ NFRが「指標 / 目標値 / 測定方法」まで定量化されているか？
□ 主要な設計判断がADRとして、トレードオフ込みで残っているか？
□ データ整合・同時実行・部分失敗の扱いが設計されているか？
□ APIの主要エラー・冪等性・互換性方針が明記されているか？
□ 例外・ログ設計で、握りつぶしと情報漏えいが避けられているか？
□ 未決事項が「要確認（誰に）」で可視化されているか？
□ 過剰設計（YAGNI違反）が無いか？

NOがある項目は、対応するフェーズのプロンプトで補強する。
```
