# プロンプト集｜HTMLツールのソースコードから「新人向け・中堅向け技術仕様書（Word）」を再現する

> 🐰🦺 面白きこともなき世を面白く（高杉晋作）
> HTMLツールの **ソースコードを渡す → 解析 → 新人向け／中堅向けの技術仕様書を Word(.docx) で生成** までを、いつでも同じ体裁で再現するためのプロンプト集です。

## このプロンプト集の前提と使い方

- **入力**：仕様化したいHTMLツールの**ソースコード全文**（単一HTMLファイル想定）。
- **出力**：同じツールの技術仕様書を、**新人向け**と**中堅向け**の2冊、**Word(.docx)** で。各冊は「仕様（概要）＋アーキテクチャ＋要件定義／詳細設計」を1冊に統合。
- **使い方**：フェーズ順に実行。`{波かっこ}` を置き換え、ソースは指定箇所に貼り付けます。最後のフェーズ3でWord生成コードまで出します。

> 💡 ポイント：**解析（フェーズ1）→ 内容生成（フェーズ2）→ Word化（フェーズ3）** の三段で回すと品質が安定します。新人向けと中堅向けは、フェーズ2で“同じ解析結果から書き分け”ます。

## フェーズ0：体裁の宣言（最初に固定）

> **ねらい**：以降の出力ブレを防ぐため、書式・作法を先に固定します。新人/中堅どちらの会話でも冒頭に置きます。

```text
これから、あるHTMLツールの「技術仕様書」を作ります。以降は次のルールを守ってください。

# ドキュメント方針
- 1冊に「第1部 仕様（概要）／第2部 アーキテクチャ仕様／第3部 要件定義／第4部 詳細設計」を統合。
- ですます調。見出し構成・表・コードブロックを積極的に使う。
- 事実（コードから確認できること）と、所感・推測を必ず分ける。推測には「※推測」を付ける。
- ソースコードに無いことを断定しない。不明点は「要確認（コード外）」と明記する。
- 固有名・捏造統計は使わない（ウソなく）。
- 後でWord(.docx)に変換するので、表は列名つき、コードはコードブロックで出す。

# 読み手（このあと指定します）
{新人エンジニア / 中堅エンジニア のどちらか}
```

## フェーズ1：ソースコード解析

> 解析結果は、新人向け・中堅向け両方の土台になります。**1回解析して使い回す**のが効率的です。

### 1-1. 全体構造の把握

```text
次のHTMLツールのソースコードを、一流エンジニアの視点で解析してください。

# ソースコード
{ここにHTMLソース全文を貼り付け}

# 出力（表・コードブロックを使う）
## 1. 全体構造
- 種別（スタンドアロン/フラグメント/SPA）、ビルド要否
- 主要セクションのツリー、外部依存（CDN/API/フォント/ライブラリ）
## 2. データ構造
- 中心データオブジェクトの完全スキーマ（プロパティ・型・必須可否）
- 初期化方法と格納場所
## 3. 状態管理
- 状態変数の一覧（名前・型・役割）と変化フロー
## 4. 関数一覧
- 全関数のシグネチャと、呼び出し関係のマップ
## 5. スタイル
- CSS変数の一覧と用途、主要コンポーネントのクラスと役割
事実のみ。読み取れない箇所は「要確認」と書く。
```

### 1-2. データフロー・永続化・書き出しの抽出

```text
同じソースコードについて、次を抽出してください。

# ソースコード（再掲）
{ソース全文}

# 出力
## イベント→状態→DOM 変化マップ（表）
| ユーザー操作 | 呼ばれる関数 | 変化する状態 | 更新されるDOM |
## 永続化
- localStorage / sessionStorage のキー一覧と用途（フォールバックの有無も）
## 書き出し・入出力
- 対応する出力形式（md/csv/json/docx/xlsx/html/印刷 等）と、その生成関数
## レンダリング戦略
- 全体再描画 と 部分更新 の条件・関数
```

### 1-3. AI/API連携の抽出（該当する場合）

```text
このツールにAI/API連携があれば、完全仕様化してください（無ければ「なし」と明記）。

# ソースコード（再掲）
{ソース全文}

# 出力
- エンドポイント・メソッド・モデル名・ヘッダー
- リクエスト構造（固定パラメータ／プロンプトのテンプレート変数）
- 認証（キーの保持場所・送信先の限定・出力への含めない方針）
- レスポンス処理（パース・成功/失敗時のUI）
- 接続テスト等の補助機能
```

### 1-4. 要件・設計の“逆抽出”

> **ねらい**：コードから、このツール自体の要件定義／設計を起こします。

```text
解析結果をもとに、このツール「自体」の要件定義と設計を逆抽出してください。

# 出力
## 要件定義
- 背景・目的（事実と目的を分ける）
- スコープ（やる/やらない）
- 機能要件FR：表 | ID | 要件 | 優先度 | 受入基準(GWT) |（コードの挙動から起こす）
- 非機能要件NFR：性能/可用性/セキュリティ/保守性/運用（測れる観点で）
## 設計
- アーキテクチャ概要（層・部品・責務）
- 主要モジュール／データ／API／例外・ログ
- 設計判断ADR（コードから読み取れる方針があれば。推測には「※推測」）
事実ベース。コードに無い数値は作らない。
```

## フェーズ2：レベル別に内容を生成

> フェーズ1の解析結果を使い、**同じ事実から新人向け／中堅向けを書き分け**ます。

### 2-A. 新人向け 本文を書く

```text
フェーズ1の解析結果をもとに、新人エンジニア向けの技術仕様書「本文」をMarkdownで書いてください。

# 構成
1. はじめに（この仕様書の読み方）
2. 第1部 仕様：このツールは何か／対象環境(表)／技術スタック(表)／外部依存／主な機能
3. 第2部 アーキテクチャ：全体の組み立て／データの流れ(表)／画面構成／保存のしくみ(表)／AI連携(任意)
4. 第3部 要件定義：背景・目的／スコープ(表)／機能要件FR(表・GWT)／非機能要件(表)
5. 第4部 詳細設計（やさしめ）：データ構造(主要キー)／主要な処理／書き出し形式(表)／ランタイムAPI例
6. 付録：用語ミニ辞典(表)／つまずきポイント

# 書き方ルール（新人向け）
- ですます調。専門用語は初出で1行補足。
- 難しい概念には「※このたとえは正確ではありません」付きのたとえを1つ。
- 完璧主義を促さない。読み拾いできる構成に。
- 事実ベース。推測には「※推測」。コードに無い数値は作らない。
```

### 2-B. 中堅向け 本文を書く

```text
フェーズ1の解析結果をもとに、中堅エンジニア向けの技術仕様書「本文」をMarkdownで書いてください。

# 構成
1. 概要と前提（設計原則）
2. 第1部 仕様：対象環境・技術スタック(表)／外部依存／設計原則
3. 第2部 アーキテクチャ：全体アーキ(テキスト図)／データフロー(表)／レンダリング戦略／状態と永続化(表)／拡張・モード分岐／AI連携仕様(表)／ランタイムサーフェス(API一覧+コード)
4. 第3部 要件定義：背景・目的／スコープ(表)／FR(表・GWT)／NFR(定量：指標/目標値/測定方法)
5. 第4部 詳細設計：データ構造(TypeScript風スキーマ)／中核処理の仕様+擬似コード／書き出し仕様(表)／例外・ログ・可観測性
6. 第5部 品質保証：テスト観点(表)／（実績があれば）テスト件数
7. 付録：拡張ポイント

# 書き方ルール（中堅向け）
- 再実装・拡張・レビューに耐える密度。トレードオフと“未決事項”を明示。
- NFRは定量化（指標/目標値/測定方法）。根拠の無い値は「要合意（誰と）」。
- 事実と所感を分離。推測には「※推測」。
```

## フェーズ3：Word(.docx)生成（スタイル仕様つき）

> **ねらい**：フェーズ2の本文を、決まった体裁のWordにします。**このスタイル仕様ブロックごと**プロンプトに渡すと、毎回同じ見た目で再現できます。

```text
次の「本文」を、下の「出力体裁スペック」に従って Word(.docx) として生成する Node.js コードを書いてください。
ライブラリは docx（npm install -g docx）。新人向け・中堅向けの2冊を別ファイルで出力できる構成にしてください。

# 本文（Markdown）
{フェーズ2-A または 2-B の出力を貼り付け}

# 出力体裁スペック（厳守）
- 用紙：A4（幅 11906 × 高 16838 DXA）、余白 上下左右 1080 DXA。
- 既定フォント：Arial（日本語はWordのフォールバックに任せる。Noto Sans JPは使わない）。本文 11pt 相当(size:22)。
- 見出し（TOC用に outlineLevel を必ず設定）：
  - H1：size 30・太字・色 9B3D6E・下に金の罫線（color C9A24E）
  - H2：size 25・太字・色 C2455C
  - H3：size 22・太字・色 5E3A52
- 表：width/columnWidths を DXA で両方指定（合計＝内容幅 9740）。
  ヘッダ行＝背景 9B3D6E＋白文字＋太字、本文は偶数行に薄い背景 FBF6F9（ゼブラ）。
  セル余白 上下60/左右110。罫線 color D8C9A0。ShadingType.CLEAR を使う（SOLID禁止）。
- コードブロック：等幅(Consolas size 18)・背景 F5F2F7・左に金の太罫線(color C9A24E)。1行＝1段落。
- ノートボックス：💡ポイント(背景 FFF4DE)／⚠️注意(背景 FDE8E8)／✅推奨(背景 E8F5EC)。左罫線つき。
- 箇条書き：LevelFormat.BULLET の numbering を使う（手打ちの「•」禁止）。
- 表紙：ツール名・【新人向け or 中堅向け】・タイトル・サブ・対象ファイル名・バージョン・作成者・タグライン。PageBreak で改ページ。
- 目次：TableOfContents（headingStyleRange "1-2"）。
- ヘッダ：右寄せでツール名＋版数、下に細い罫線。フッタ：中央にタグライン＋ページ番号(PageNumber.CURRENT)。

# 注意（docx-jsの落とし穴）
- 改行に \n を使わず、段落を分ける。
- 段落の境界(border)は top→left→bottom→right の順。コードブロックは「左罫線のみ」にして順序エラーを避ける。
- 表は width を必ず DXA で（PERCENTAGE禁止）。columnWidths の合計＝表幅に一致させる。
- PageBreak は Paragraph の中に入れる。
```

> 💡 ポイント：上の「出力体裁スペック」は、新人向け・中堅向けで**共通**です。本文だけ差し替えれば、2冊が同じ見た目で揃います。

## フェーズ4：検証とレンダリング確認

```text
生成した .docx を検証し、問題があれば直してください。

# 手順
1. 検証：python scripts/office/validate.py 出力.docx（"All validations PASSED!" を確認）
2. 体裁確認：.docx → PDF に変換し、表紙・目次・表のヘッダ色・コードブロック・ノートボックス・ページ番号を目視
3. よくある不具合と対処：
   - 境界要素の順序エラー → コードブロックは左罫線のみにする
   - 表が崩れる → columnWidths と各セル width の両方をDXAで設定、合計を一致
   - 日本語が出ない → 既定フォントArialのままでOK（Wordが日本語をフォールバック）。文字化け時はエンコード(UTF-8)確認
4. 直したら再検証 → PASSしたら完成
```

## レベル別 書き分けガイド

| 観点 | 新人向け | 中堅向け |
| --- | --- | --- |
| 目的 | 全体像をやさしく理解 | 再実装・拡張・レビューに耐える |
| 用語 | 初出で1行補足＋たとえ(※不正確) | 既知前提。深い用語をそのまま |
| 要件 | 主要FRを抜粋・一言メモ付き | FR是正＋NFRを定量(指標/目標/測定) |
| 設計 | データ構造は主要キーのみ | 完全スキーマ(TypeScript風)＋擬似コード |
| 追加章 | 用語辞典・つまずきポイント | レンダリング戦略・トレードオフ・品質保証・拡張点 |
| トーン | 完璧主義を促さない | 未決事項と割り切りを明示 |

## チェックリスト

```text
仕様書(Word)が完成したら、次を確認してください。

【内容】
□ コードに無いことを断定していないか（推測に「※推測」）？
□ FRの受入基準がGWTで書けているか？
□ NFRが（中堅向けは）定量化されているか？
□ データ構造・主要関数・書き出し・永続化・AI連携を網羅したか？

【体裁(Word)】
□ validate.py が PASS したか？
□ 表紙・自動目次・ヘッダ/フッタ・ページ番号が出ているか？
□ 表のヘッダ色／ゼブラ／コードブロックの金罫線／ノートボックスが反映されているか？
□ 新人向け・中堅向けで“同じ見た目・違う中身”になっているか？
```
