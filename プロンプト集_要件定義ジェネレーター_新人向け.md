# プロンプト集【新人向け】｜要件定義ジェネレーターで“設計書こわい”を卒業する

> 🐰🦺 面白きこともなき世を面白く（高杉晋作）
> 「うさうさ開発フロア PRO（要件定義＆詳細設計ジェネレーター）」に貼り戻して使う、**新人エンジニア向け**のAIプロンプト集です。専門用語のミニ解説と「つまずきポイント」つき。

## この集の使い方（3ステップ）

1. プロンプトの **コードブロックをコピー** します。
2. `{波かっこ}` を自分の課題の言葉に置き換えます。
3. Claude などに貼って実行し、出てきた文章を **ツールのフォーム欄に貼り戻します**。

> 💡 新人のうちは「一発で完璧」を狙わなくて大丈夫です。**ドラフトを出す → 直す** のくり返しが、結局いちばん早く上達します。

## 0. 用語ミニ辞典（まずこれだけ）

| 用語 | 1行でいうと |
| --- | --- |
| 要件定義 | 「何を作るか・なぜ作るか」を決めて書くこと |
| 詳細設計 | 「どう作るか」を、誰が作っても同じになるよう書くこと |
| FR（機能要件） | 「○○できること」という、できるべき機能の一つひとつ |
| NFR（非機能要件） | 速さ・安全・使いやすさなど「どれくらいの品質か」 |
| GWT | 受入基準の書き方。Given（前提）/ When（操作）/ Then（結果） |
| ADR | 「なぜその方式を選んだか」を残す設計判断メモ |
| トレーサビリティ | 要件とテストがちゃんと結びついているか、の追跡 |

> ### 🍜 たとえ話：うさうさラーメン店（※このたとえは正確ではありません）
> 要件定義＝お客様の注文票、基本設計＝店のレイアウト、詳細設計＝レシピ。
> 注文票が曖昧だと、厨房が優秀でも違う料理が出てきます。まずは「何が欲しいか」をはっきりさせましょう。

## フェーズ1：全体像をつかむ

### 1-1. 「このタスク、結局なにを作るの？」を整理する

> **ねらい**：頭の中のモヤモヤを、背景・目的・作るものに分けてもらいます。

```text
あなたは新人エンジニアを指導する先輩です。
私が任されたタスクを、新人にも分かるように整理してください。

# 私が聞いている話
{聞いたこと・作りたいものを、わかる範囲で箇条書き}

# お願い
- ですます調で、専門用語には初出で1行の補足を付けてください。
- 「現状こまっていること（事実）」「達成したい状態（目的）」「今回作るもの」の3つに分けてください。
- 私がこの先「確認すべきこと」を、誰（役職名）に聞けばよいか付きでリストにしてください。
- 推測の部分には「※たぶん」と付けてください。
```

### 1-2. 埋める順番をナビしてもらう

> **ねらい**：ツールのどの欄から手を付ければいいか、迷わないようにします。

```text
これから「要件定義書 兼 詳細設計書」を書きます。
次の内容をもとに、初心者が迷わない「書く順番」を提案してください。

# 作るもの
{作るものの概要}

# お願い
- 「まず書くべき欄 → 次 → …」の順で、各ステップ1行の理由つき。
- 各ステップで「最低これだけ書けばOK」という合格ラインも添える。
- 完璧主義にならないよう、後回しでよい欄も教えてください。
```

## フェーズ2：要件を書く

### 2-1. 背景・目的を書く

```text
次の機能について「背景・目的」を書いてください。

# 機能のこと
{機能の説明}

# お願い
- ですます調・200〜300字。
- 「困っていること（事実）」と「こうしたい（目的）」を分ける。
- 根拠のない数字は書かない。わからない数字は「（要確認）」と置く。
```

### 2-2. スコープ（やること／やらないこと）を切る

```text
次の機能の「やること」と「やらないこと」を表にしてください。

# 機能
{機能の概要}

# お願い
- 「スコープ内」「スコープ外」の2列。
- 迷うものは「要確認」に分け、誰（役職名）に聞くか書く。
- 新人がつい盛り込みがちな“余計な機能”があれば、やらないこと側に寄せて理由を1行。
```

### 2-3. 機能要件（FR）を GWT で起こす ★最重要

> **ねらい**：そのままテストになる形で機能を書きます。ここが書ければ半分終わりです。

```text
次の機能を、機能要件（FR）の一覧にしてください。

# 機能
{機能の説明}

# お願い（形式を必ず守ってください）
- 各要件に「FR-01」のような連番ID。
- 各要件に優先度（Must / Should / Could / Won't）。
- 受入基準を必ず GWT で：
  Given（前提）/ When（操作）/ Then（期待結果）
- 表：| ID | 要件 | 優先度 | 受入基準(GWT) |
- 1つの要件に機能を詰め込みすぎない。テストできる最小単位に割る。
- 初心者向けに、各FRに「これは何のための要件か」を一言メモも付ける。
```

### 2-4. 非機能要件（NFR）をやさしく洗い出す

```text
次のシステムについて、非機能要件（NFR）の“たたき台”を作ってください。

# システム
{概要}

# お願い
- カテゴリ：速さ / 安全 / 使いやすさ / 動き続ける（可用性）/ 直しやすさ。
- 各項目、まずは「どんな観点があるか」を新人にわかる言葉で。
- 数値は決め打ちせず、「ここはチームに確認」と書いてOK。
```

## フェーズ3：設計を書く

### 3-1. 基本設計（全体の組み立て）をやさしく

```text
次の要件をもとに、基本設計の“全体像”をやさしく書いてください。

# 機能要件（FR）
{FR一覧を貼り付け}

# お願い
- 「どの部品が、何をするか」を表（層 / 部品 / 役割）で。
- データの流れ（入力 → 処理 → 出力）を3〜5行で。
- 専門用語には1行の補足。難しい判断は「ここは先輩に相談」と印を付ける。
```

### 3-2. 設計判断（ADR）を1件、書いてみる

```text
次の「どっちにするか迷っていること」について、ADRを1件書いてください。

# 迷っていること
{例：データの保存先を A にするか B にするか}

# お願い
- 「論点 / 決定 / 理由 / 他の案」の4項目。
- 理由は、事実（要件・制約）と、好み（所感）を分けて書く。
- 新人向けに、各案の「メリット・デメリット」も1行ずつ添える。
```

## フェーズ4：テストとレビュー

### 4-1. FR からテスト観点を作る

```text
次の機能要件（FR）から、テスト観点を作ってください。

# 機能要件
{FR一覧（IDと受入基準GWT）を貼り付け}

# お願い
- 「区分（正常 / 異常 / 境界）/ 観点 / 対応FR」の表。
- すべてのFRが、最低1つのテスト観点から参照されるように。
- GWTの Then を、具体的な確認手順に書き直す。
```

### 4-2. 自分の設計書をやさしくレビューしてもらう

```text
私が書いた設計書を、新人にも分かるようにレビューしてください。
ダメ出しだけでなく、直し方とよい点も教えてください。

# 設計書
{ツールの .md 出力を貼り付け}

# お願い
- 「曖昧な言葉」「抜けている観点」「テストで確認できない要件」を指摘。
- 各指摘に、直し方の例を1つ添える。
- よい点も3つ以上。きびしくしすぎない。
- 重要なものから順に。
```

## 新人あるある：つまずきポイント集

- **「全部完璧に埋めなきゃ」と固まる** → まず Must の要件と背景だけでOK。残りは「要確認」で前に進めましょう。
- **FRに機能を詰め込みすぎる** → 「○○して××して△△する」は3つに分割。1FR＝1つの確認できること。
- **受入基準が書けない** → GWTのテンプレ（Given/When/Then）に当てはめるだけ。埋まらない＝要件が曖昧、というサインです。
- **NFRの数字が決められない** → 決めなくてOK。「（要確認・誰に）」と書けば立派な仕事です。
- **AIの出力をそのまま信じる** → AIは下書き役。最後は自分の目で、たとえ話の「※不正確」と同じく疑ってかかる癖を。

## チェックリスト（やさしめ）

```text
設計書を書いたら、次の問いに答えられるか確認しましょう。

□ この機能は「何のため」かを1文で言えるか？
□ Must の機能要件（FR）は、GWTで受入基準が書けているか？
□ 各FRに、対応するテスト観点があるか？（ツールの帯が緑か）
□ 「やらないこと」を1つ以上書いたか？
□ わからない所を「要確認（誰に）」で残せているか？
□ AIの出力を、自分の言葉で説明できるか？

ぜんぶ△以上なら、最初の設計書として合格です🐰
```
